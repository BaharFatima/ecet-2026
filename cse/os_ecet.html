Operating System Concepts

What is an Operating System?

An OS is a software that acts as an intermediary between user applications and computer hardware, managing resources and providing services.

Goals of Operating System:

  • Resource Management: Allocate and manage hardware resources (CPU, memory, I/O)
  • Convenience: Provide easy-to-use interface for users
  • Efficiency: Maximize resource utilization
  • Security: Protect data and resources from unauthorized access
  • Reliability: Ensure stable and predictable operation

Services Provided by OS:

  • Program Execution: Load and execute programs
  • I/O Operations: Handle input/output operations
  • File System Management: Create, manage, and access files
  • Process Management: Create, schedule, and terminate processes
  • Error Detection: Detect and handle errors
  • Resource Allocation: Allocate resources among competing processes
  • Accounting: Keep track of resource usage

Types of Operating Systems:

Type Features Examples Batch OS Jobs processed in batches without user interaction, high throughput IBM mainframe systems Time Sharing OS Multiple users share system resources, interactive UNIX, Linux Real-time OS Guaranteed time to respond to external events, hard/soft deadlines VxWorks, QNX Distributed OS Multiple processors/machines working together MINIX, Amoeba Network OS Connected computers share resources over network Windows Server, Linux

System Calls:

Definition: Interface between user-level programs and kernel, allowing processes to request OS services

Categories of System Calls:

  • Process Control: fork(), exec(), exit(), wait()
  • File Management: open(), close(), read(), write(), seek()
  • Directory Management: mkdir(), rmdir(), chdir()
  • Information Maintenance: getpid(), getppid(), time()
  • I/O Control: ioctl(), read(), write()
  • Communication: pipe(), socket(), send(), receive()
  • Protection: chmod(), chown()
Common MCQ Pattern:
  • Identify which service is NOT provided by OS
  • Difference between goals and services
  • Purpose of a specific system call
  • Type of OS for specific scenario (real-time vs batch)

Process Management

What is a Process?

A process is an instance of a program in execution. It includes program code, data, stack, heap, and the execution state of the program.

Process Control Block (PCB):

PCB is a kernel data structure that contains all information about a process.

PCB Contains:

  • Process ID (PID): Unique identifier for the process
  • Program Counter (PC): Address of next instruction to execute
  • CPU Registers: Accumulator, general-purpose registers, stack pointer
  • Process State: Current state (running, waiting, ready, etc.)
  • Priority: Priority level for scheduling
  • Memory Pointers: Base and limit registers, page tables
  • I/O Status: List of I/O devices allocated, open files
  • Accounting Information: CPU time used, memory used, creation time

Process States:

State Description Transition New Process is being created в†’ Ready (when resources allocated) Ready Process is waiting for CPU to execute в†’ Running (when scheduler selects it) Running Process is executing on CPU в†’ Ready (time quantum ends) or Waiting (I/O request) Waiting Process waiting for I/O or event в†’ Ready (when I/O completes) Terminated Process execution completed Final state

Threads:

Thread is a lightweight process - the smallest unit of execution. A process can have multiple threads sharing the same memory space.

Process vs Thread:

Aspect Process Thread Creation Time High (slow) Low (fast) Context Switch Expensive (save/restore entire PCB) Cheap (only save/restore registers) Memory Separate memory space Share memory with other threads Communication IPC mechanisms needed Direct shared memory access Synchronization Less needed Critical (shared resources)

Types of Threads:

  • User-level Threads: Managed by user library, kernel not aware
    • Fast context switching
    • Blocking system call blocks entire process
  • Kernel-level Threads: Managed by OS kernel
    • Slower context switching
    • One thread's I/O wait doesn't block others
  • Hybrid Model: Combination of user and kernel threads

CPU Scheduling

CPU Scheduling Criteria:

  • CPU Utilization: Keep CPU as busy as possible (40-90%)
  • Throughput: Number of processes completed per unit time (maximize)
  • Turnaround Time: Time from submission to completion (minimize)
  • Waiting Time: Time spent in ready queue (minimize)
  • Response Time: Time from submission to first output (minimize)
  • Fairness: All processes get fair CPU time
Turnaround Time = Completion Time - Arrival Time Waiting Time = Turnaround Time - Burst Time Response Time = Time when first execution starts - Arrival Time

CPU Scheduling Algorithms:

1. First Come First Served (FCFS)

Concept: Processes execute in the order they arrive. Queue-based.
  • Pros: Simple, fair, no starvation
  • Cons: Poor average waiting time, convoy effect (short jobs wait for long jobs)
  • Best for: Batch systems
Example: P1(24ms), P2(3ms), P3(3ms) arriving in order
Timeline: P1(0-24), P2(24-27), P3(27-30)
Avg Waiting = (0 + 24 + 27)/3 = 17ms

2. Shortest Job First (SJF)

Concept: Process with smallest burst time executes first. Can be preemptive or non-preemptive.
  • Pros: Minimizes average waiting time (optimal for non-preemptive)
  • Cons: Can starve long processes, difficult to predict burst time
  • Preemptive SJF: Shortest Remaining Time First (SRTF)
Example: P1(24ms), P2(3ms), P3(3ms)
Timeline: P2(0-3), P3(3-6), P1(6-30)
Avg Waiting = (6 + 0 + 3)/3 = 3ms

3. Round Robin (RR)

Concept: Each process gets a fixed time quantum (time slice). If not complete, goes to back of queue.
  • Pros: Fair, good response time, no starvation, suitable for time-sharing
  • Cons: Higher context switching overhead, average waiting time depends on quantum
  • Time Quantum: 10-100ms (critical parameter)
Time Quantum Effect:
вЂў Too small: More context switches, more overhead
вЂў Too large: Behaves like FCFS, poor response time
вЂў Ideal: Just enough for 80% of processes to complete in one quantum

4. Priority Scheduling

Concept: Each process has a priority. Higher priority processes execute first.
  • Types: Preemptive (higher priority preempts) or Non-preemptive
  • Pros: Important processes handled first
  • Cons: Starvation of low-priority processes (solution: aging)
Aging: Gradually increase priority of waiting processes to prevent starvation

5. Multilevel Queue Scheduling

Concept: Processes divided into classes (foreground/background, system/user). Each class has separate queue with its own scheduling algorithm.
  • Fixed priority between queues (can lead to starvation)
  • Time sharing between queues

6. Multilevel Feedback Queue Scheduling

Concept: Processes can move between queues based on behavior. Most flexible and complex.
  • Process moves to lower priority queue if uses too much CPU time
  • Process moves to higher priority queue if waiting too long (aging)
  • Parameters to define: Number of queues, scheduling algorithm for each, promotion/demotion rules
MCQ Tip: MLFQ is most commonly used in modern OSes (Unix, Linux, Windows) because it provides good performance for mixed workloads.
Common MCQ Pattern:
  • Calculate waiting time, turnaround time for given algorithms
  • Compare average waiting times of different algorithms
  • Identify which algorithm minimizes response time / turnaround time
  • Starvation vs fairness trade-off
  • Effect of time quantum on performance
  • Context switch overhead comparisons
  • Real-time OS scheduling (not preemptable vs preemptable)

Inter-Process Communication & Synchronization

Inter-Process Communication (IPC):

IPC is the mechanism to allow processes to exchange data and synchronize execution. Critical for cooperating processes.

Methods of IPC:

  • Pipes: One-way, unnamed, used for parent-child communication
  • Named Pipes (FIFO): Bidirectional, named, can be used by unrelated processes
  • Shared Memory: Fast, but requires explicit synchronization
  • Message Queues: Asynchronous communication
  • Sockets: Network communication between processes
  • Signals: Asynchronous notification to process

Process Synchronization Problem:

Problem: When multiple processes access shared resources concurrently, results depend on execution order (race condition). Must ensure mutual exclusion.

Critical Section Problem:

Critical Section: Code segment where shared resources are accessed. Only one process should execute at a time.

Requirements for Solution:

  • Mutual Exclusion: Only one process in critical section at a time
  • Progress: Process not in critical section cannot block others
  • Bounded Waiting: Limit on how many times other processes can enter before given process

Semaphores:

Semaphore is an integer variable accessed only through two atomic operations: wait() and signal().

wait(S): signal(S): while S<=0 S++ ; if (process waiting) S-- wake up process

Binary Semaphore:

  • Value is 0 or 1 (like a mutex/lock)
  • 0 = locked, 1 = unlocked
  • Used for mutual exclusion

Counting Semaphore:

  • Value can be any non-negative integer
  • Represents number of available resources
  • Used for resource pools
Semaphore for Mutual Exclusion:
semaphore mutex = 1; process P_i: wait(mutex); [critical section] signal(mutex);

Problems with Semaphores:

  • Incorrect use can lead to deadlock or race conditions
  • Not suitable for complex synchronization patterns

Monitors:

Monitor is a higher-level synchronization construct. It's a class with synchronized methods. Only one process can execute monitor method at a time.

Monitor Features:

  • Automatically handles mutual exclusion
  • Condition variables for complex synchronization
  • No need for explicit wait/signal calls
  • Cannot be interrupted inside monitor
Semaphore vs Monitor: Monitor is safer, more structured; Semaphore is lower-level, more flexible.
Common MCQ Pattern:
  • Purpose of wait() and signal()
  • Binary vs counting semaphore
  • Race condition detection
  • Critical section problem solution
  • Monitor advantages over semaphores
  • Deadlock with semaphores

Deadlocks

What is Deadlock?

A deadlock is a situation where a set of processes are blocked permanently, waiting for resources held by other processes in the set.

Example: P1 holds R1, waiting for R2. P2 holds R2, waiting for R1. Both blocked forever.

Necessary Conditions for Deadlock:

All four conditions must be true simultaneously:

  • Mutual Exclusion: Resource cannot be shared (only one process at a time)
  • Hold and Wait: Process holds resources while waiting for others
  • No Preemption: Resources cannot be forcibly taken away
  • Circular Wait: Chain of processes, each holding resource needed by next

Deadlock Handling Strategies:

1. Deadlock Prevention

Break one of the four necessary conditions:

  • Mutual Exclusion: Not always possible (some resources inherently non-shareable)
  • Hold and Wait: Force process to acquire all resources before starting or wait for none (inefficient resource utilization)
  • No Preemption: Preempt resources from process if needed (difficult to implement)
  • Circular Wait: Order resources linearly; process requests in increasing order (practical approach)
Resource Ordering: Define ordering R1 < R2 < R3... Process must request in this order. Prevents circular wait.

2. Deadlock Avoidance

Idea: Don't allocate resources if it could lead to unsafe state. Requires knowing future resource demands.

Banker's Algorithm:

  • Maintains available, maximum, allocated matrices for each resource and process
  • Before allocating, check if state remains safe
  • Safe state: Sequence of processes exists where each can acquire needed resources
  • Advantage: More efficient than prevention
  • Disadvantage: Processes must declare max resource needs in advance
Available = Total - Allocated Need = Maximum - Allocated Safe State Check: 1. Find process P_i with Need[i] <= Available 2. If found: Available += Allocated[i], repeat 3. If all processes complete: SAFE, else UNSAFE

3. Deadlock Detection

Idea: Allow deadlocks to occur, but detect and recover.

Detection Algorithm:

  • Maintain resource allocation graph or matrices
  • Periodically check for circular wait
  • If detected, perform recovery

Detection using Resource Allocation Graph:

  • Process в†’ Resource: Process waiting for resource
  • Resource в†’ Process: Resource allocated to process
  • Cycle in graph = Deadlock (if single instance per resource)

4. Deadlock Recovery

Actions to break deadlock once detected:

  • Process Termination: Abort one or more deadlocked processes
    • Abort all deadlocked processes
    • Abort one at a time until cycle broken
  • Resource Preemption: Forcibly take resources from processes
    • Select victim (minimize cost)
    • Rollback process to safe state
    • Restart process
Strategy Approach Overhead Use Case Prevention Break one condition High (resource utilization poor) Simple systems, strict guarantee needed Avoidance Check before allocating (Banker's) Medium (checking overhead) Known resource demands Detection Allow, then detect and recover Medium (detection + recovery) Deadlocks rare; recovery cost acceptable Ignorance Assume deadlocks won't occur Low Most practical systems (Linux, Windows)
Common MCQ Pattern:
  • Identify necessary conditions from scenario
  • Which condition can be broken for deadlock prevention?
  • Safe vs unsafe state detection
  • Banker's algorithm feasibility check
  • Circular wait detection from allocation graph
  • Recovery strategy for given deadlock
  • Comparison of strategies

Memory Management

Memory Management Concepts:

OS must allocate memory to processes, manage shared memory, and ensure protection between processes.

Memory Allocation Schemes:

1. Overlays

  • Keep in memory only the parts of program needed at any time
  • Other parts stored on disk
  • Manually managed by programmer
  • Used when physical memory < logical memory needed

2. Swapping

Swapping: Moving entire process between main memory and disk (swap space).

  • Swap out: Move process to disk when not immediately needed
  • Swap in: Bring process back to memory when needed
  • Reduces memory pressure, enables more processes
  • Expensive operation (I/O bound)

3. Fragmentation

Fragmentation: Wasted memory that cannot be allocated to processes.

External Fragmentation:

  • Free memory split into small scattered pieces
  • Total free memory sufficient, but no contiguous block large enough
  • Caused by allocate/deallocate operations

Internal Fragmentation:

  • Allocated memory contains unused space
  • Caused by allocation unit size (e.g., page size)
  • Less than one allocation unit wasted per process
Solution for External Fragmentation:
вЂў Compaction: Move all processes together (expensive)
вЂў Paging: Divide memory into fixed-size pages (solves external, introduces internal)

4. Paging

Paging: Divide physical memory into fixed-size frames; logical memory into pages. Pages can be scattered in frames.

Paging Concepts:

  • Page: Fixed-size block in logical memory (e.g., 4KB)
  • Frame: Fixed-size block in physical memory (same size as page)
  • Page Table: Maps logical page number to physical frame number
  • Page Table Entry: Contains frame number + control bits (valid, protection, etc.)
Logical Address = Page Number + Offset Physical Address = Frame Number + Offset Page Number = Logical Address / Page Size Offset = Logical Address % Page Size
Advantages: Eliminates external fragmentation
Disadvantage: Internal fragmentation (on average half page wasted per process)

5. Segmentation

Segmentation: Divide logical memory into variable-size segments (code, data, heap, stack). Physical memory allocated per segment.

  • Segment Table: Maps segment number to base address and limit
  • Logical address = (segment number, offset within segment)
  • Advantage: Logical memory organization, supports variable-size allocation
  • Disadvantage: External fragmentation
Aspect Paging Segmentation Block Size Fixed (all pages same size) Variable (each segment different) External Fragmentation No Yes Internal Fragmentation Yes (on average 0.5 page) No Table Size Large (one entry per page) Small (one entry per segment) Protection Per page Per segment (better)
Common MCQ Pattern:
  • Logical to physical address translation
  • Calculate fragmentation given memory state
  • Page table size, number of entries
  • Internal vs external fragmentation identification
  • Comparison of schemes
  • Address space calculation

Virtual Memory

What is Virtual Memory?

Virtual Memory: Technique allowing process to use memory addresses larger than physical memory. Illusion of unlimited memory to each process.

Benefits:

  • Run processes larger than physical memory
  • Multiple processes in memory simultaneously
  • Easier to write programs (don't worry about memory limits)
  • Protection between processes

Virtual Memory Implementation:

  • Demand Paging: Load pages only when needed
  • Demand Segmentation: Load segments only when needed (less common)

Demand Paging:

Idea: Keep only frequently used pages in memory. Bring others from disk on demand.

Page Fault:

  • Occurs when process references page not in memory
  • CPU raises page fault exception
  • OS page fault handler loads page from disk

Page Fault Handling Steps:

  1. Check if page reference is valid
  2. If invalid: Terminate process (segmentation fault)
  3. If valid but not in memory:
    • Find free frame
    • Schedule I/O to read page
    • Update page table
    • Restart instruction
Effective Access Time:
EAT = (1 - p) Г— Memory Access Time + p Г— Page Fault Time Where p = probability of page fault Memory Access Time в‰€ 100 nanoseconds Page Fault Time в‰€ 25 milliseconds (disk access)

Example: p = 0.001, Memory=100ns, PageFault=25ms
EAT = 0.999Г—100 + 0.001Г—25000 = 99.9 + 25 = 124.9ns (25% slower!)

Page Replacement Algorithms:

When memory full and new page needed, must decide which page to remove.

1. FIFO (First In First Out)

  • Replace oldest page (first one loaded)
  • Pros: Simple to implement
  • Cons: Often removes frequently used pages
  • Anomaly: Increasing frames can increase page faults (Belady's Anomaly)

2. LRU (Least Recently Used)

  • Replace page not used for longest time
  • Pros: Better performance than FIFO, reasonable results
  • Cons: Complex implementation (need to track access time), overhead

3. Optimal

  • Replace page that will not be used for longest time in future
  • Pros: Minimum page faults (theoretically optimal)
  • Cons: Cannot predict future, only used as comparison benchmark
Algorithm Complexity Performance Belady's Anomaly FIFO O(1) Poor Yes LRU O(n) or O(1) with hardware Good No Optimal N/A (theoretical) Best No

Thrashing:

Thrashing: CPU spends more time page faulting than executing. Severe performance degradation.

Causes:

  • Too many processes in memory
  • Working set of processes exceeds physical memory

Working Set Model:

  • Working Set: Set of pages actively used by process
  • Working Set Size (WSS): Number of distinct pages in working set
  • Sum of WSS for all processes should be в‰¤ physical memory
  • If sum > physical memory: Thrashing

Prevention:

  • Suspend processes if sum of WSS > available memory
  • Allow only processes whose WSS fits
  • Monitor page fault rate, throttle when high
Common MCQ Pattern:
  • Calculate effective access time
  • Page fault handling steps
  • Compare page replacement algorithms
  • Belady's anomaly identification
  • Thrashing detection and prevention
  • Working set calculation
  • Page fault vs hit determination

Disk Scheduling

Disk I/O Performance:

Disk I/O is major bottleneck. OS must schedule requests to minimize average access time.

Disk Access Time Components:

Total Access Time = Seek Time + Rotational Delay + Transfer Time Seek Time: Time to move disk head to correct track (major factor, 5-10ms) Rotational Delay: Time for sector to rotate under head (avg 0.5 rotation, 2-4ms) Transfer Time: Time to transfer data (usually negligible)

Disk Scheduling Algorithms:

1. FIFO (First In First Out)

  • Requests served in order of arrival
  • Pros: Simple, fair
  • Cons: Lots of head movement, high average seek time
Example: Queue: 98, 183, 37, 122, 14, 124, 65, 67
Head starts at 53
Movements: 53в†’98в†’183в†’37в†’122в†’14в†’124в†’65в†’67
Total seek distance: (98-53) + (183-98) + (183-37) + (122-37) + (122-14) + (124-14) + (124-65) + (67-65) = 640

2. SSTF (Shortest Seek Time First)

  • Serve request with minimum seek time from current head position
  • Pros: Less head movement than FIFO, more efficient
  • Cons: Can starve requests far from current position, greedy algorithm
Example: Queue: 98, 183, 37, 122, 14, 124, 65, 67
Head at 53: Closest is 65 (12)
в†’ 65: Closest is 67 (2)
в†’ 67: Closest is 98 (31)
в†’ 98: Closest is 122 (24)
в†’ 122: Closest is 124 (2)
в†’ 124: Closest is 183 (59)
в†’ 183: Closest is 37 (146)
в†’ 37: Closest is 14 (23)
Total = 299 (much better than FIFO!)

3. SCAN (Elevator Algorithm)

  • Head moves in one direction, serving all requests, then reverses
  • Pros: Reduces starvation, uniform service, popular in practice
  • Cons: Bias against middle tracks (serve more frequently)

4. C-SCAN (Circular SCAN)

  • Head moves in one direction only, returns to start without serving
  • More uniform wait times than SCAN
  • Pros: Fair service, no bias
  • Cons: More head movement than SCAN
Algorithm Head Movement Fairness Best Use FIFO High Good Light load SSTF Medium Poor (starvation) Not recommended SCAN Low Good High load, general purpose C-SCAN Medium Best Real-time systems
Common MCQ Pattern:
  • Calculate total seek distance for given algorithm
  • Compare algorithms for given request queue
  • Identify which algorithm minimizes starvation
  • Disk access time components
  • Why SCAN is preferred over SSTF
  • Seek time vs rotational delay

File Management

File Concept:

File: Named collection of related information recorded on secondary storage. Logical unit of information.

File Operations:

  • Create: Allocate space, create directory entry
  • Open: Validate access, load into memory
  • Read: Read data from current position
  • Write: Write data at current position
  • Seek: Move file pointer to desired position
  • Delete: Deallocate space, remove directory entry
  • Close: Release file handle, flush buffers
  • Append: Add data at end of file
  • Rename: Change file name

File Access Methods:

1. Sequential Access

  • Read/write data in order, one after another
  • File pointer moves automatically
  • Suitable for: Magnetic tapes, batch processing

2. Direct (Random) Access

  • Read/write any block without reading previous blocks
  • Must specify block number or offset
  • Suitable for: Disk files, databases

3. Indexed Sequential Access

  • Combination of sequential and direct access
  • Index structure allows direct access to key positions
  • Suitable for: Large data files with frequent searches

Directory Structure:

Directory: Special file containing file names and inode/FCB references.

Single-level Directory:

  • All files in single directory
  • Simple but inflexible
  • File name conflicts possible

Two-level Directory:

  • Master File Directory (MFD) for each user
  • User File Directory (UFD) for each user's files
  • Prevents name conflicts, but still inflexible

Hierarchical (Tree-structured) Directory:

  • Root directory with subdirectories
  • Full path name: /dir1/dir2/.../filename
  • Current working directory concept
  • Most common (Unix, Windows, etc.)

Acyclic Graph Directory:

  • Allows shared directories (hard links)
  • More flexible than tree (one parent allowed)
  • Complexity: Ensure no cycles

General Graph Directory:

  • Allows cycles (symbolic links)
  • Most flexible but complex
  • Garbage collection needed

File Allocation Methods:

1. Contiguous Allocation

  • File occupies contiguous blocks on disk
  • Pros: Fast access, simple, efficient for sequential access
  • Cons: External fragmentation, difficult to expand files
  • Used in: CD-ROMs, DVDs

2. Linked Allocation

  • Each block contains pointer to next block
  • Pros: No external fragmentation, flexible
  • Cons: Slow access (must follow pointers), pointer overhead, unreliable (lost pointer = lost data)
  • Improvement: FAT (File Allocation Table) - keep pointers in separate table

3. Indexed Allocation

  • Index block contains pointers to all blocks of file
  • Pros: Direct access, no external fragmentation
  • Cons: Index overhead, multilevel indexing for large files
  • Used in: Most modern systems (Unix inodes, NTFS)
Method Access Speed Space Efficiency Fragmentation Contiguous Fastest Poor (external) External Linked Slow Good (with FAT) None Indexed Medium Good None

Free Space Management:

  • Bit Vector: One bit per block (1=free, 0=allocated) - simple but large
  • Linked List: Free blocks linked together - slow but space-efficient
  • Grouping: Store groups of free blocks
  • Counting: Store (first free block, count of consecutive free blocks)
